Morphing Liquid Nav

Four navigations, one element. Pill · Tabs · Chips · Segmented. They're not different components — they're the same nav, morphing. The persistent coral indicator travels across shape changes via the View Transitions API, something that was impossible in CSS until 2024.

click states · arrows to navigate · ⌘1–⌘4 to morph
    state pill · active Overview

    One DOM element. The same .liquid-nav transforms its padding, gap, radius, and background per state. The indicator travels via view-transition-name.

    Drop-in. The whole idea in one file. One <nav>, four CSS classes that reshape it, and document.startViewTransition() to morph between them. Copy, open, morph.

<script>
  function stitMorph(state) {
    const nav = document.getElementById('stit-mln');
    function apply() {
      nav.className = 'stit-mln is-' + state;
    }
    if (document.startViewTransition) document.startViewTransition(apply);
    else apply();
  }
  // ⌘1–⌘4 to morph
  document.addEventListener('keydown', (e) => {
    if (!(e.metaKey || e.ctrlKey)) return;
    const map = { '1': 'pill', '2': 'tabs', '3': 'chips', '4': 'segmented' };
    if (map[e.key]) { e.preventDefault(); stitMorph(map[e.key]); }
  });
</script>

    Apple-Craft

    Shared-element morphs are the signature of Apple's design language — iOS app switcher, the Dynamic Island, macOS Spotlight. This component brings that language to the web.

    Editorial-Display

    One element that reveals different faces based on context is editorial at its purest: the same sentence told in prose, verse, dialogue, and bullet. Form adapts, meaning persists.

    Power-Dense

    ⌘1–⌘4 to switch navigation mode is power-user candy. A single surface that adapts to the task at hand reduces cognitive load even when it increases density.

    Playful-Warm

    The morph itself is delightful. Watching a pill stretch into tabs and pour into chips is the kind of detail that makes users smile on first discovery and lean into the product.

    • ✕
      Cross-fading between four separate nav components

      Separate <PillNav>, <TabsNav>, <ChipsNav>, and <SegmentedNav> components, swapped with a fade on mode change. Users see four things dying and being born. The unity is lost.

      Instead: one element. Change its class. Let the browser animate the delta through View Transitions. It is the same element — treat it as such.

    • ✕
      Morphing just to show off

      Using this component when a plain set of tabs would do. The morph should be meaningful — different states have different affordances (multi-select, compact view, mobile form-factor).

      Instead: morph only when states serve different needs. Pill = collapsed dropdown. Tabs = focused single selection. Chips = multi-filter. Segmented = mode switch.

    • ✕
      Using a JS physics library for the morph

      Framer Motion, GSAP, Popmotion. 80 KB of JavaScript for something the browser can now do natively.

      Instead: document.startViewTransition(() => apply()). Three lines. Native. Battery-friendly. Progressive enhancement — falls back to instant change on unsupported browsers.

    • ✕
      Letting the indicator fade out and re-appear

      The indicator dies on the old state and is re-instantiated on the new. The user loses the thread of "which item is active" during the transition.

      Instead: give the indicator a stable view-transition-name. It becomes a shared element — the browser animates its position and shape across the morph, never losing it.

    • ✕
      Forgetting prefers-reduced-motion

      A 520ms spring morph is delightful for most. For a vestibular-sensitive user it's nausea. Autoplay without consent is accessibility failure.

      Instead: wrap the startViewTransition call in a check. Fall back to an instant class change if reduced motion is requested.
